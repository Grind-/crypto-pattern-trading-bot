Crypto Pattern AI

Claude-gesteuertes Backtesting & Live-Trading-System für USDC-Kryptowährungspaare auf Binance.

Python 3.12 FastAPI Chart.js 4.4 Docker Compose Binance API Anthropic Claude

Projektübersicht

Crypto Pattern AI nutzt Claude als KI-Analyst, um technische Chartmuster in historischen Kerzendaten zu erkennen, daraus Handelsstrategien abzuleiten, diese per Backtesting zu validieren und schließlich automatisiert auf Binance live zu handeln.

  • Simulation: Claude analysiert bis zu 1000 Kerzen mit 8 technischen Indikatoren, generiert BUY/SELL-Signale, der Simulator berechnet P&L inkl. Gebühren — iterativ bis zu einer profitablen Strategie.
  • Live Trading: Wartet auf Kerzen-Schlusskurs (UTC-aligned), scannt alle konfigurierten Paare, wählt das beste, fragt Claude nach einem Signal und führt Marktorders auf Binance aus.
  • Multi-User: Vollständig isolierte Datenbereiche pro User — eigene Simulationen, eigener Live-State, eigene Claude-Credentials.

Datenfluss

Simulations-Flow

1
Daten laden: fetch_klines(symbol, interval, days) → bis zu 1000 OHLCV-Kerzen von Binance
2
Indikatoren: compute_indicators(candles) → 8 Indikator-Spalten auf jedem Datenpunkt
3
Claude-Analyse: Gesampelte Indikatortabelle (max. 80 Zeilen) → Claude generiert BUY/SELL-Signale + Strategiename + Konfidenz
4
Simulation: run_simulation(candles, signals) → P&L, Win-Rate, Drawdown, Gebühren, Portfolio-Historie
5
Feedback-Loop: Nicht profitabel → Feedback (Fehlschläge) an Claude für nächste Iteration. Max. N Iterationen.
6
Persistenz: Bestes Ergebnis → save_simulation(username, entry, full_result) auf Disk

Live-Trading-Flow

1
Kerzenschluss berechnen: UTC-alignierter nächster Schlusskurs + 10s Buffer → sleep
2
Marktscan: Alle 10 + konfigurierten Extra-Paare gleichzeitig fetchen → Claude rankt und wählt bestes Paar
3
Symbol-Wechsel: FLAT → wechsle zu bestem Paar; IN_POSITION → bleibe beim aktuellen Symbol
4
Signal: get_live_signal() mit letzten 100 Kerzen + optional Strategie-Kontext + Signal-Historie
5
Ausführung: BUY bei Konfidenz ≥ 60 % (FLAT); SELL bei Konfidenz ≥ 55 % (IN_POSITION) → Binance Market-Order
6
State-Update: Position, Trade-History, Buy-Preis → in-memory + periodisch auf Disk (Restart-Sicherheit)

Simulator

Die Backtesting-Engine in simulator.py berechnet realistische P&L inkl. Gebühren.

  • Strikte Alternierung: Nur BUY→SELL→BUY-Sequenzen; offene BUY-Positionen am Ende werden zum Schlusskurs mit Gebühr geschlossen.
  • Gebühren: Werden auf Kapital (BUY) und Erlös (SELL) separat abgezogen. Fee-Drag = Gesamtgebühren / Startkapital.
  • Net P&L: Vergleicht Netto-Erlös nach Sell-Fee mit originalem Kapital vor Buy-Fee (echter Return).
  • Drawdown: Rollendes Maximum des Portfolio-Werts; Drawdown = (Peak − Current) / Peak.
  • Portfolio-Historie: Snapshot nach jeder Kerze (für Chart-Darstellung).

Marktscanner

Vor jedem Kerzenschluss im Live-Loop werden alle konfigurierten Paare parallel gescannt.

  • Standard-Paare (10): BTC, ETH, BNB, SOL, XRP, ADA, AVAX, DOGE, DOT, LINK — alle USDC
  • Extra-Paare (bis zu 3): Vom User frei konfigurierbar (per localStorage gespeichert)
  • Parallel-Fetch: Alle Symbole werden gleichzeitig über asyncio.gather() geladen (60 Kerzen)
  • Claude-Ranking: Kompakte Tabelle mit Preis, 24h/7d-Änderung, ATR%, RSI, MACD-Richtung, Volumen-Ratio → Claude vergibt 0–100 Punkte
  • Symbol-Wechsel: Nur bei FLAT möglich. Bei IN_POSITION wird nur informiert, gewechselt wird nach dem nächsten SELL.

Live-Loop Details

Zeitalignierung

Der nächste Kerzenschluss wird UTC-aligniert berechnet:

next_close = (int(now / interval_seconds) + 1) * interval_seconds
wake_at    = next_close + 10  # 10s Buffer nach offiziellem Schluss

Der Loop schläft in 30s-Chunks damit live_state["running"] = False prompt erkannt wird.

Konfidenz-Schwellwerte

AktionMinimale KonfidenzBedingung
BUY≥ 60 %Nur wenn Position FLAT
SELL≥ 55 %Nur wenn Position IN_POSITION
HOLD—Immer möglich, keine Order

Symbol-Tracking

  • current_symbol: Aktuell bevorzugtes Paar (kann sich bei FLAT ändern)
  • position_symbol: Gesperrtes Paar sobald eine BUY-Order platziert wird; wird erst nach SELL freigegeben
  • active_symbol: position_symbol wenn IN_POSITION, sonst current_symbol

Auth & Rollen

AspektDetail
SessionsIn-Memory Dict token → {username, expiry}. Token = 64-char Hex via secrets.token_hex(32). TTL: 7 Tage. Werden bei Neustart zurückgesetzt (Re-Login erforderlich).
PasswörterPBKDF2-SHA256 mit Salt "cpa_salt_bioval_2026", 200.000 Iterationen. Nur Hash in users.json gespeichert.
MiddlewareAlle Pfade außer /login, /auth/*, /static/* prüfen Session-Cookie. APIs geben 401 JSON zurück; Seiten → Redirect zu /login.
Rolle: adminZugang zu /admin und allen /api/admin/*-Endpoints. Kann user anlegen/löschen/sperren, Platform-Zugang vergeben.
Rolle: userZugang zu eigenen Daten. Kann eigenen API-Key / OAuth-Token konfigurieren, Passwort ändern. Kein Platform-Zugang ohne Admin-Freigabe.
DatenisolationAlle API-Routes extrahieren Username aus Session; Simulation und Live-State sind per-User getrennt (in-memory + on-disk).

Sicherheitshinweise

Claude API-Keys und OAuth-Tokens werden aktuell im Klartext in users.json gespeichert. Für Produktionsumgebungen sollte eine Verschlüsselung (z.B. Fernet) ergänzt werden.
Binance API-Keys werden für Restart-Persistenz in live_state.json gespeichert. Die Datei liegt auf dem Host-Volume und ist nur root-lesbar.
Empfohlene Binance API-Berechtigungen: Spot Trading (Enable Spot & Margin Trading). Keine Auszahlungsrechte vergeben.
