Crypto Pattern AI

Claude-gesteuertes Backtesting- und Live-Trading-System für USDC-Kryptowährungspaare auf Binance. Claude analysiert Chartmuster, generiert BUY/SELL-Signale — der Rest läuft automatisch.

Python 3.12 · FastAPI Vanilla JS · Chart.js 4.4 Docker Compose Binance Spot API Claude Sonnet 4.6

✦Kernfunktionen

Drei integrierte Subsysteme arbeiten zusammen — vom Backtesting bis zur automatischen Orderausführung.

🔬
Backtesting / Simulation
Claude analysiert bis zu 1000 historische Kerzen mit 11 Indikatoren und generiert BUY/SELL-Signale. Der Simulator berechnet P&L inkl. Gebühren. Nicht profitable Strategien gehen mit Feedback zurück an Claude — iterativ bis zu einer profitablen Lösung.
📡
Live Trading
Wartet auf UTC-alignierte Kerzenschlüsse, scannt parallel alle konfigurierten Paare, wählt das beste Symbol und fragt Claude nach einem Signal. Marktorders werden automatisch auf Binance ausgeführt.
👥
Multi-User-System
Vollständig isolierte Datenbereiche pro User — eigene Simulationen, eigener Live-State, eigene Claude-Credentials. Admin-Panel zur Benutzerverwaltung, drei Claude-Zugangsmodi.

⬡Systemarchitektur

Zwei Docker-Container im selben Netzwerk. Der Hauptcontainer enthält FastAPI + alle Geschäftslogik; der Proxy-Container kapselt die Claude CLI für Platform/Subscription-Modus.

⚙Backend-Module (app/)

Jede Datei hat eine klar abgegrenzte Verantwortung. Kein Framework-Overhead — nur Python-Standardbibliotheken + FastAPI + httpx + pandas.

FastAPI-App, alle Routes, Auth-Middleware, per-User In-Memory-State (sim_states, live_states), Background Task Orchestrierung, Auto-Resume bei Start
user_store.py
Vollständiges User-CRUD auf users.json: Anlegen, Löschen, Sperren, Passwortverwaltung, Claude-Konfiguration pro User. PBKDF2-SHA256 Hashing.
claude_analyst.py
Alle Claude-Aufrufe: Backtesting-Analyse, Live-Signal, Markt-Scan, Verbindungstest. Dispatcht zwischen Proxy (Platform/Subscription) und direkter API (API-Key).
indicators.py
Technische Indikatorberechnung auf pandas DataFrame: RSI, MACD, Bollinger Bands, SMA/EMA, ATR, StochRSI, Volume Ratio, Price Changes.
simulator.py
Backtesting-Engine: strikte BUY→SELL-Alternierung, P&L inkl. Gebühren auf beiden Seiten, rollendes Drawdown, Portfolio-Historie für Chart-Darstellung.
binance_trader.py

▷Simulations-Flow

Eine Simulation iteriert zwischen Claude-Analyse und Backtesting, bis eine profitable Strategie gefunden oder das Maximum an Iterationen erreicht ist.

1
Kerzen laden: fetch_klines(symbol, interval, days) — bis zu 1000 OHLCV-Kerzen von Binance (/api/v3/klines).
2
Indikatoren: compute_indicators(candles) — 11 Indikatoren werden auf jedem Datenpunkt der pandas-DataFrame berechnet (RSI, MACD, Bollinger etc.).
3
Claude-Analyse: Gesampelte Indikatortabelle (max. 80 Zeilen) → Claude antwortet mit Strategiename, Analyse, erkannten Mustern, BUY/SELL-Signalen mit Confidence.
4
Backtesting: run_simulation(candles, signals, fee_pct) — berechnet P&L, Win-Rate, Drawdown, Gebühren, Portfolio-Historie für jeden Trade.
5
Feedback-Loop: Nicht profitabel? Fehlgeschlagene Trades werden als Feedback an Claude übergeben. Nächste Iteration mit angepasster Strategie. Max. N Iterationen (User-konfigurierbar).
6
Speichern: Bestes Ergebnis → save_simulation(username, ...) — Metadaten in simulations.json, Vollresultat (Trades, Signale, Candles) in sims/{id}.json.

▶Live-Trading-Flow

Der Live-Loop läuft als FastAPI Background Task und synchronisiert sich auf UTC-alignierte Kerzenschlüsse.

1
Zeitalignierung: next_close = (int(now/interval_s)+1) * interval_s + 10s — 10s Buffer nach offiziellem Kerzenschluss. Loop schläft in 30s-Chunks (damit Stop-Requests sofort greifen).
2
Marktscan: Alle 10 Standard-Paare + bis zu 3 User-definierte Extras werden gleichzeitig über asyncio.gather() gefetcht. Claude rankt alle Paare nach Handelspotenzial (0–100 Punkte).
3
Symbol-Wahl: Bei FLAT → wechsle zum besten Paar. Bei IN_POSITION → bleibe beim position_symbol (das gelockte Paar beim letzten BUY). Wechsel nur nach SELL.
4
Signal: get_live_signal() mit 100 Kerzen + aktuelle Position + letzten 10 Signalen + optionaler Strategie-Kontext aus einer gespeicherten Simulation.
5
Ausführung: BUY wenn Confidence ≥ 60 % und FLAT. SELL wenn Confidence ≥ 55 % und IN_POSITION. Orders werden als Binance Market-Orders platziert (quoteOrderQty).
6
State-Update: Position, Trade-History, Buy-Preis → in-memory + live_state.json auf Disk. Ermöglicht Wiederaufnahme nach Container-Neustart (_auto_resume_all()).

Konfidenz-Schwellwerte

AktionMin. ConfidenceBedingung
BUY≥ 60 %Position muss FLAT sein
SELL≥ 55 %Position muss IN_POSITION sein
HOLD—Kein Schwellwert; keine Order wird ausgeführt

◯Claude-Zugangsmodi

Drei Wege, Claude zu nutzen — konfigurierbar pro User in den Einstellungen.

Platform
Server-Account
Nutzt den OAuth-Token des Servers. Kein eigener Key nötig. Wird vom Admin freigeschaltet. Route: claude-proxy → claude CLI mit CLAUDE_CODE_OAUTH_TOKEN des Servers.
Subscription
Eigene claude.ai-Sub
User hinterlegt eigenen OAuth-Token (sk-ant-oat01-…) aus ~/.claude/.credentials.json. Proxy überschreibt den Server-Token per-Request durch den User-Token.
API-Key
Eigener API-Key
User hinterlegt sk-ant-api03-… Key. Direkte HTTP-Anfrage an api.anthropic.com/v1/messages — kein Proxy involviert. Pay-per-use.

∿Technische Indikatoren

Berechnet in indicators.py mit pandas/numpy. Alle Indikatoren werden vor der Übergabe an Claude auf jedem Datenpunkt berechnet.

RSI Periode 14 MACD EMA 12/26/9 Bollinger Bands SMA 20 · 2σ SMA 20 / 50 EMA 12 / 26 ATR Periode 14 StochRSI Periode 14 Volume Ratio vs. SMA-20-Vol Price Change 1 / 4 / 12 Kerzen
IndikatorInterpretation
RSI< 30 = überverkauft (Kaufsignal); > 70 = überkauft (Verkaufssignal)
MACD / HistogrammPositiv + steigend = bullish; negativ + fallend = bearish. Histogramm zeigt Momentum-Divergenz.
bb_pct (Bollinger)0 = Preis am unteren Band (überverkauft); 1 = Preis am oberen Band (überkauft)
ATRVolatilitätsmaß in Preis-Einheiten. Hoher ATR = große Schwankungen → engere Stops empfehlenswert.
StochRSIRSI des RSI, normiert 0–1. Sensitiver als RSI, reagiert früher auf Trendwenden.
Volume Ratio> 1.5 = überdurchschnittliches Volumen (bestätigt Bewegung); < 0.5 = schwaches Signal.

🔒Auth & Rollen

Sitzungs-basierte Authentifizierung, vollständig in-memory. Keine externe Datenbank, kein JWT.

AspektDetail
SessionsIn-Memory Dict token → {username, expiry}. Token: 64-char Hex (secrets.token_hex(32)). TTL: 7 Tage. Verfallen bei Container-Neustart.
PasswörterPBKDF2-SHA256, 200 000 Iterationen. Nur Hash in users.json.
MiddlewareAlle Pfade außer /login, /auth/*, /static/* prüfen Cookie. APIs → 401 JSON; Seiten → Redirect zu /login.
Rolle adminZugang zu /admin, /docs, allen /api/admin/*. Kann User anlegen, sperren, löschen, Platform-Zugang vergeben.
Rolle userZugang nur zu eigenen Daten. Kann API-Key/OAuth-Token konfigurieren, Passwort ändern. Kein Platform-Zugang ohne Admin-Freigabe.
DatenisolationAlle Routes holen Username aus Session; State-Dicts und Disk-Pfade sind vollständig per-User getrennt.

💾Persistenz & Restart

Das Host-Volume ./data → /app/data überlebt Container-Neustarts. Beim Start ruft _auto_resume_all() alle User mit was_running=true wieder auf.

DatenPersistiertRestart-Verhalten
Simulationsergebnisse✓ JaVollständig erhalten; werden beim Start nicht neu geladen.
Live-State (war aktiv)✓ Ja_auto_resume_all(): alle User mit was_running=true werden automatisch wiederaufgenommen.
User-Datenbank✓ Jausers.json bleibt erhalten; Admin wird nur einmalig beim ersten Start angelegt.
In-Memory Sessions✗ NeinAlle Sessions verfallen → Re-Login erforderlich.
Laufende Simulationen✗ NeinWerden abgebrochen. Bereits gespeicherte Ergebnisse bleiben.

⚠Sicherheitshinweise

Claude API-Keys und OAuth-Tokens werden im Klartext in users.json gespeichert. Für höhere Sicherheit kann eine Verschlüsselung (z.B. Fernet/AES) ergänzt werden.
Binance API-Keys werden für Restart-Persistenz in live_state.json gespeichert. Die Datei liegt auf dem Host-Volume (nur root-lesbar).
Empfohlene Binance-Berechtigungen: Nur Spot Trading (Enable Spot & Margin Trading). Keine Auszahlungs- oder Transferrechte.
Sessions: In-Memory — nicht zwischen Instanzen teilbar und nicht persistent. Multi-Instance-Deployments würden ein externes Session-Store (Redis o.ä.) erfordern.
